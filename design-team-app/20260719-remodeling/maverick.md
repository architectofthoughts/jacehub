# jacehub 리모델링 — 모험파(ad-maverick) 제안

2026-07-19 · 근거: `app.js`(1,924줄)·`lobby.js`·`quests.js`·`functions/api/*`·`schema.sql`·퀘스트 설계 spec 직접 확인

## 코드에서 읽은 판 (제안의 전제)

- **index가 사실상 '전 함대 재고표'다.** 프로젝트 전수 그리드 + stats + radar + 필터/정렬 + CSV 내보내기 — 다중 사용자 SaaS 대시보드의 문법을 1인용 도구가 그대로 이식했다. 사용자 1인에게 이 그리드는 매일 거의 같은 그림, 즉 정보가 아니라 벽지다.
- **radar가 이미 '오늘 뭘 봐야 하는지'를 계산하고 있다** (`getProjectRadar`·`getRadarOverview`, attention/stale/fresh). 그런데 그 결과를 또 전체 그리드 위에 얹어 보여줄 뿐, 행동으로 이어주지 않는다.
- **lobby는 토큰리스지만 대시보드가 같은 기기에서 먼저 돌아야 산다** (`jacehub_lobby_cache` localStorage 스냅샷). 폰의 새 브라우저에서 lobby만 열면 빈 화면 — "런처"라면서 관제탑에 인질로 잡혀 있다.
- **quests.html은 관리자 CRUD 폼이다.** 습관화가 목적인 시스템의 얼굴이 `<select>` 4개짜리 등록 폼 + 잔디 한 장. 동기 부여 장치가 UI에 거의 없다.
- **파이프는 이미 훌륭하다.** ingest→quest_events→daily_progress, 순수 로직 분리, PIN 단일 인증. 아래 제안 전부 이 파이프를 재사용하며, 데이터 마이그레이션이 필요한 안은 없다.

---

## 제안 1 — 허브 전복: '오늘(Today)'이 정문, 대시보드는 격납고로

**무엇을**: 첫 화면을 갈아엎어 단일 'Today' 스크린으로. 구성은 위에서부터 ① 어제 이후 바뀐 것만 보여주는 diff 피드(새 배포·실패 전환·stale 진입 — radar 신호 재사용) ② 오늘의 퀘스트 요약 ③ 즐겨찾기 런처 상단 한 줄. 현재의 전수 그리드+필터+CSV는 '격납고(Hangar)' 탭으로 강등해 한 겹 뒤로 보낸다. Today 피드는 서버(D1 스냅샷 diff)에서 만들어 lobby의 localStorage 인질 문제도 함께 해소.

**깨는 관습**: "대시보드는 전체 현황을 항상 보여줘야 한다." 다중 사용자 SaaS에서 온 가정이다. 1인용 관제탑에서 매일 필요한 건 전체가 아니라 *변화량*이다. (레퍼런스: Linear의 Triage — 이슈 전체 목록이 아니라 '판단 필요한 것'만 정문에 세운다.)

**성공 시 UX**: 폰에서 열고 10초 안에 "밤새 뭐 바뀌었고 오늘 뭐 할지" 끝. 3페이지 왕복이 소멸하고, index·lobby·quests의 존재 이유가 각각 diff·실행·동기로 재정렬된다.

**최소 검증**: 기존 스냅샷 캐시만 읽는 `today.html` 한 장을 만들어 1주간 첫 진입을 그리로 리다이렉트 — 격납고를 '굳이 다시 여는' 횟수가 주 2회 이하면 전복 확정.

## 제안 2 — 함대 오버월드: 허브를 도구가 아니라 게임 화면으로

**무엇을**: lobby+quests를 합쳐 하나의 '마을 오버월드'로 리스킨. 앱 = 건물(배포 이력·나이가 건물 레벨), 데일리 퀘스트 = 마을 게시판의 퀘스트 로그, 스트릭 = 버프 아이콘, ingest 이벤트 = 경험치. 핵심 한 방: **방치된 프로젝트는 건물이 시각적으로 낡아간다** — radar의 stale 수치를 숫자가 아니라 잡초와 금 간 벽으로 번역한다.

**깨는 관습**: "생산성 도구는 도구처럼 생겨야 한다." Habitica가 습관 앱을 RPG로 뒤집어 살아남았고, Duolingo가 출석부를 게임 루프로 바꿔 카테고리를 재정의했다. 그리고 이 사용자는 게임 디렉터에, 이미 여행 준비를 RPG 플레이스루로 만드는 사람이다 — 함대-마을 은유는 남의 옷이 아니라 본인의 모국어다.

**성공 시 UX**: 퀘스트 체크가 '출석 의무'에서 '마을 돌보기'로 바뀐다. 방치 앱이 낡아가는 모습은 radar 배지보다 강한 회귀 신호가 되고, 허브 접속 자체가 데일리 루프가 된다.

**최소 검증**: 백엔드 무변경 Wizard of Oz — 스냅샷+`/api/progress`만 읽는 `overworld.html`(SVG 마을) 한 장을 옆에 세워 1주간 quests.html 대비 자발적 재방문 횟수를 비교.

## 제안 3 — 읽는 허브에서 시키는 허브로: 카드에 동사(⚡)를 단다

**무엇을**: 프로젝트 카드가 지금은 '보기'뿐이다. 여기에 액션 동사를 단다 — 대표로 ⚡ 버튼: 누르면 crossbell 깨우기 웹훅(이미 3머신 실측 완료된 기존 인프라)에 프로젝트명+한 줄 지시를 실어 에이전트에게 작업을 발주한다. "youtubewc 실패 배지 확인 → 같은 카드에서 ⚡ '배포 실패 원인 봐줘'" 가 한 흐름이 된다.

**깨는 관습**: "대시보드는 읽기 전용, 조치는 터미널에서." 상태를 보는 곳과 고치는 곳이 분리돼 있다는 가정은 에이전트 함대 시대에 이미 낡았다. (레퍼런스: GitHub의 'assign issue to Copilot' — 이슈 화면이 곧 발주 창구가 된 순간, PagerDuty 모바일의 원탭 ack.)

**성공 시 UX**: 허브가 뷰어에서 리모컨으로. 폰에서 문제를 *발견*한 자리에서 *해결 발주*까지 끝나므로, 관제탑이라는 이름이 처음으로 사실이 된다. 성공하면 동사를 늘린다(재배포, 로그 열기, 리뷰팀 소집).

**최소 검증**: Fake Door 반 발짝 위 — 카드 한 장에만 ⚡를 달고 기존 hermes 웹훅으로 POST하는 20줄 패치, 1주간 누른 횟수와 발주 후 실제 처리 완료율을 센다.

## 제안 4 — 퀘스트 발행권 역전: 내가 아니라 함대가 퀘스트를 낸다

**무엇을**: 유저가 CRUD 폼으로 등록하는 습관 퀘스트 위에, **시스템이 매일 발행하는 함대 퀘스트** 레인을 추가한다. 소스는 허브가 이미 아는 것들 — radar의 attention/stale 신호, 실패 배포, (확장 시) improvement-plan 잔여 항목. 매일 아침 3개 한도로 D1 `quests`에 자동 INSERT(발행일 스탬프), 클리어는 기존 ingest(배포 이벤트 등) 또는 원탭 수동 체크.

**깨는 관습**: "습관 앱은 사용자가 목표를 입력하는 곳이다." 1인 도그푸딩에서 입력 폼 자체가 마찰인데, 정작 '오늘 뭘 해야 하는지'의 데이터는 허브가 전부 쥐고 있다. 입력자와 수혜자가 동일인일 때 입력은 낭비다. (레퍼런스: Duolingo 데일리 퀘스트 — 유저는 한 번도 퀘스트를 '작성'하지 않는다.)

**성공 시 UX**: quests 페이지가 '내가 관리하는 할 일 목록'에서 '함대가 올리는 아침 미션 브리핑'으로 반전된다. radar 숫자가 처음으로 행동 단위로 번역되고, 제안 2(오버월드 게시판)와 결합하면 완성형.

**최소 검증**: 컨시어지 MVP — 서버 코드 0줄, 바니가 아침 루틴에서 radar JSON을 읽어 D1에 퀘스트 3개를 넣어주는 스크립트 하나로 1주 운영, 시스템 퀘스트 클리어율 ≥ 습관 퀘스트 클리어율이면 정식 구현.

## 조합 제언과 선 긋기

- 네 안은 축이 다르다: **1=정보구조 반전, 2=표현 반전, 3=방향 반전(읽기→쓰기), 4=발행 주체 반전.** 최대 조합은 1+4를 골격으로, 2를 스킨으로, 3을 카드 액션으로 얹는 것 — 다만 검증은 각각 따로, 위의 값싼 방법으로 먼저.
- 일부러 안 낸 것: 프레임워크 도입, 3페이지 정리·통합 같은 정돈, PIN UX 개선, 성능/캐시 손질. 필요하지만 판을 흔드는 일이 아니다 — 정합파(ad-canon)의 몫으로 남긴다.
- 공통 안전핀: 어느 안도 VAULT·DB 바인딩, ingest 계약, 기존 데이터를 건드리지 않는다. 전부 파이프 위의 새 얼굴이라 실패해도 회수 비용이 화면 한 장이다.
