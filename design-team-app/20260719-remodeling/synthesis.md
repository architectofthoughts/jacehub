## 총론 — 두 파가 같은 곳을 찌른 지점

제안 8개는 A/B 대결이 아니다. 진단이 세 곳에서 겹친다:

1. **로비의 localStorage 인질** — 정합2와 모험1이 같은 문제를 다른 층위에서 푼다 (KV 승격 vs 서버 diff 피드). → 축 2
2. **퀘스트 방의 빈약함** — 정합4(품질 회수)와 모험4(발행권 역전)는 같은 방을 고치되 상호 배타가 아니다. → 축 4
3. **데일리 루프의 단절** — 정합3(로비 배지)과 모험2(오버월드)는 같은 루프의 노출 강도만 다르다. → 축 3

양 파 공통 안전핀: VAULT·DB 바인딩, ingest 계약, 기존 데이터 무변경. 아래 모든 선택지가 이 전제를 지킨다.

## 결정 축 6개

### 축 1 — 정문: 첫 진입 화면이 보여주는 것

| 선택지 | 내용 | 출처 |
|---|---|---|
| **1A 현행+복원** | 관리 대시보드 그대로, 마지막 방문 방 복원만 추가 | 정합1 일부 |
| **1B Today 스트립** | index 상단에 "어제 이후 바뀐 것 + 오늘 퀘스트 요약" 스트립, 그리드는 아래 유지 | 교배 (모험1의 축소 이식) |
| **1C Today 정문** | `today.html`이 정문, 전수 그리드는 '격납고' 탭으로 강등 | 모험1 |

- 트레이드오프: 1A는 "매일 같은 벽지" 문제 존치 / 1B는 한 화면이 길어짐 / 1C는 정보 밀도를 원할 때 한 겹 더 들어가야 함.
- 전제: 1B는 radar 신호(현행 클라 계산)만으로 가능. **진짜 "어제 이후" diff는 서버 스냅샷 이력(축2-C) 필요.**
- 되돌리기: 1A·1B 낮음(스트립 제거) / 1C 중간 — 리다이렉트 롤백은 쉬우나 동선 재학습 2회 발생.

### 축 2 — 상태의 집: 스냅샷·메타의 저장 위치 (기기 독립화)

| 선택지 | 내용 | 출처 |
|---|---|---|
| **2A 현행 유지** | localStorage 전용 — 폰 새 브라우저에서 로비 빈 화면 존치 | — |
| **2B KV 봉투 승격** | `favorites` 옆에 로비 스냅샷·메타 저장, localStorage는 오프라인 캐시로 강등 | 정합2 |
| **2C 2B + D1 스냅샷 이력** | 일일 스냅샷을 D1에 적재 → diff 피드(1C)·건물 노화(3C)의 재료 | 모험1 서버 요건의 명시화 |

- 트레이드오프: 2B는 새 개념 제로, `syncFavoritesToVault()` 패턴의 연장 / 2C는 테이블 신설(additive)이지만 적재 트리거가 미확정.
- 되돌리기: 2B 거의 0(봉투 필드 무시하면 끝) / 2C 소(테이블 방치 가능).
- 보안 전제 (양 파 공통): 스냅샷에 내부 도메인(warren 등)이 실리므로 **PIN 뒤에서만 읽기 유지** — 공개 read로 낮추지 말 것.
- **의존성: 축 1·3·4의 모험 쪽 선택지 전부와 정합 배지안까지, 사실상 전 조합의 공통 토대.**

### 축 3 — 데일리 루프의 노출 지점

| 선택지 | 내용 | 출처 |
|---|---|---|
| **3A 현행** | 퀘스트 상태는 quests.html에만 | — |
| **3B 타일 배지** | 로비(+index 퀵 로비) 타일에 미완 수 배지 — 미완일 때만, 설정 토글로 방어 | 정합3 |
| **3C 오버월드 리스킨** | lobby+quests를 마을 화면으로 — stale=건물 노화, 스트릭=버프 | 모험2 |

- 조합 가능: **3B와 3C는 택일이 아니다** — 3B는 본선에 지금, 3C는 별동 `overworld.html`(Wizard of Oz, 백엔드 무변경)로 병행 실험 가능.
- 트레이드오프: 3B는 학습 비용 0(iOS 배지 규약)이나 회귀 신호가 약함 / 3C는 회귀 신호가 강하고 사용자 모국어(게임 은유)이나 에셋 제작·유지 비용이 큼.
- 되돌리기: 3B 낮음 / 3C 실험이면 화면 한 장, 정식 채택이면 재학습+에셋 비용이 크므로 반드시 실험 경유.

### 축 4 — 퀘스트 발행 주체

| 선택지 | 내용 | 출처 |
|---|---|---|
| **4A 습관 퀘스트 품질 회수** | 토큰 승계·편집 UI(`PUT` 기존재)·전부 클리어 축하·`EVENT_CATALOG` D1화 (1차 범위는 여기까지, 스트릭 최고기록·월별 통계는 2라운드) | 정합4 |
| **4B 함대 퀘스트 레인** | radar 신호→매일 최대 3개 시스템 발행, 클리어는 ingest 또는 원탭 | 모험4 |
| **4C 병행 (4A+4B)** | 상호 배타 아님 — 습관 레인은 품질 회수, 함대 레인은 컨시어지 MVP로 검증 | 교배 |

- 트레이드오프: 4A는 확실하나 동기 부여 구조는 그대로 / 4B는 "입력 낭비" 문제를 정면으로 풀지만 발행 규칙 설계가 필요.
- 마이그레이션 (schema.sql 실측): **현행 `quests` 테이블에 발행 주체 컬럼 없음** — 4B 정식화 시 `origin` 류 additive ALTER 필요(기존 데이터 무손실, 롤백=컬럼 무시). 컨시어지 MVP 단계는 서버 코드 0줄이라 스키마도 무변경.
- 되돌리기: 4A 낮음(빈 방 채우기) / 4B 컨시어지 0(스크립트 중단), 정식화 후 회수는 발행 rows 정리 정도(소).

## 조합 규칙 (의존성)

- **2B는 전 조합의 선행 토대** — 어떤 조합이든 첫 스프린트에 포함할 것 (정합 우선순위 1 = 모험의 "인질" 해소, 양 파 합의점).
- 1C(Today 정문) ⇒ 2C 필요. 3C 정식 채택 ⇒ 6C와 세트. 5C ⇒ 5B 선행 필수. 4B 정식화 ⇒ additive 스키마 변경.
- 교배는 자유: 예) "골격은 정합, 파일럿은 모험" / "6A 골격 + 3C 오버월드만 정식" / "1C만 채택하고 3은 배지로".

## 추천 조합 — "정합 골격 + 모험 파일럿 3종"

**본선(1라운드): 2B → 3B + 4A(병행) → 6A.** 순서는 정합파 제안(2→3→4→1)을 그대로 승계.
**별동 파일럿(본선과 병행, 각 1주 계측): 1B Today 스트립 · 4B 컨시어지 MVP · 5B ⚡ 1카드.**
**2라운드 결정 보류: 1C·2C·3C·6C** — 파일럿 지표(격납고 재방문 주 2회 이하? 함대 퀘스트 클리어율 ≥ 습관? ⚡ 사용·완료율?)를 보고 판단.

근거: ① 사용자의 구체 불만 지점이 미확정인 상태에서 되돌리기 비용이 가장 큰 축(6C)을 먼저 걸지 않는다. ② 정합 4안은 전부 기존 API·패턴의 빈 방 채우기라 실패 리스크가 최소인데, 모험 4안은 전부 값싼 검증 경로를 스스로 제시했다 — 둘을 순차가 아니라 **병렬**로 돌리면 본선 골격을 훼손하지 않고 반전 가설을 전부 계측할 수 있다. ③ 전 항목이 데이터 보존·바인딩 유지 제약을 충족한다.

## 미확정 (지어내지 않은 것들)

- 사용자의 구체 불만 지점 (packet 명시 — 이 문서의 축 선택으로 표출될 것)
- 3페이지 유지 여부 최종 판단 (축 6의 결정으로 귀결)
- 2C 스냅샷 적재 트리거 (cron vs 대시보드 방문 시)
- 1B/1C의 "변화" 정의 — diff로 칠 radar 신호 목록(새 배포·실패 전환·stale 진입 외 추가 여부)
- 4B 발행 규칙 상세 (3개 한도의 우선순위, 신호→클리어 조건 매핑)
- 5B 오발주 방지 UX·발주 상태 피드백 채널
- 3C 오버월드의 아트 볼륨과 에셋 제작 여력

## ad-techdesigner 인수 메모

목업으로 보여줘야 화면 차이가 판정되는 결정 지점: **축 1**(index 상단 스트립 vs Today 정문+격납고), **축 3**(배지 얹힌 로비 vs 오버월드 마을), **축 6**(3탭 내비 시안 — 로비 축소형 포함), **축 4B**(quests 화면의 함대 레인 1장), **축 5B**(카드 ⚡ 버튼과 발주 후 상태). 축 2와 4A는 화면 차이가 작으니 목업 없이 체크박스 결정으로 충분.
